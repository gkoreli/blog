# ADR-0004: Analytics ‚Äî Custom Lightweight Analytics on Cloudflare Workers

## Status

Accepted ‚Äî 2026-03-06

## Context

The blog needs analytics to answer: did anyone visit? Which posts get read? Where did they come from? Are they real humans or bots ‚Äî or me testing?

### Why This Matters Now

GitHub Pages provided zero website analytics ([community discussion #31474](https://github.com/orgs/community/discussions/31474) ‚Äî 441 upvotes, 55 participants, unanswered since Aug 2022). This was a key driver for migrating to Cloudflare Workers (ADR-0001).

After migrating, we enabled Cloudflare Web Analytics (built-in, free). First day results revealed its limitations:
- **43 visits, 94 page views** ‚Äî sounds impressive, but ~42 were the site owner testing during development
- **1 visit from github.com** ‚Äî the only confirmed real visitor
- **4 visits to `/bad-url`** ‚Äî the owner debugging 404 handling, counted as 4 separate "visits"
- **No way to filter yourself out** ‚Äî "visits" counts sessions, not unique people. Every new tab or return is a new visit.
- **No public dashboard** ‚Äî can't share stats transparently
- **No API** ‚Äî can't build a custom `/stats` page
- **Performance-focused** ‚Äî Core Web Vitals dominate the dashboard, content analytics are secondary

Bottom line: Cloudflare Web Analytics tells you your LCP is 180ms (great) but can't tell you if anyone actually read your article.

### What We Actually Need

- **Which posts get read** ‚Äî page views per post, ranked
- **Real unique visitors** ‚Äî with the ability to exclude the site owner
- **Referrer sources** ‚Äî GitHub, Google, Twitter, direct?
- **Geographic data** ‚Äî where are readers from?
- **Bot filtering** ‚Äî don't count crawlers
- **Public dashboard** ‚Äî transparency, fits the "Prompted by human, generated by AI" ethos
- **Data ownership** ‚Äî query it, export it, build on it

### What We Don't Need

- Funnels, conversion tracking, ecommerce attribution
- Session replay, heatmaps
- Team management, multi-site
- Real-time dashboards (daily aggregates are fine)

### Platform Capabilities (Verified from Cloudflare Docs)

Now on Cloudflare Workers, we have access to:

**Workers (free tier):**
- 100,000 requests/day, 10ms CPU time per invocation
- Requests to static assets are **free and unlimited** ‚Äî only Worker script invocations count
- Source: [Workers Pricing](https://developers.cloudflare.com/workers/platform/pricing/)

**D1 (free tier):**
- 5 million rows read/day, 100,000 rows written/day, 5 GB total storage
- Max 10 databases, 500 MB per database on free plan (10 GB on paid)
- Limits reset daily at 00:00 UTC
- No egress charges, no throughput charges
- Time Travel: 7-day point-in-time recovery (free), 30-day (paid)
- Source: [D1 Pricing](https://developers.cloudflare.com/d1/platform/pricing/), [D1 Limits](https://developers.cloudflare.com/d1/platform/limits/)

**`request.cf` object (free, server-side, all plans):**
- `country` ‚Äî two-letter country code (e.g., `"US"`)
- `city` ‚Äî city name (e.g., `"Austin"`)
- `continent` ‚Äî continent code (e.g., `"NA"`)
- `region` ‚Äî ISO 3166-2 region name (e.g., `"Texas"`)
- `timezone` ‚Äî IANA timezone (e.g., `"America/Chicago"`)
- `colo` ‚Äî IATA airport code of the Cloudflare datacenter (e.g., `"DFW"`)
- `asOrganization` ‚Äî ASN organization (e.g., `"Google Cloud"`)
- `isEUCountry` ‚Äî `"1"` if EU country
- Source: [Request `IncomingRequestCfProperties`](https://developers.cloudflare.com/workers/runtime-apis/request/#incomingrequestcfproperties)

**`run_worker_first` (routing control):**
- Accepts `true` (all requests) or `string[]` (route patterns like `["/api/*"]`)
- When set to route patterns, only matching requests invoke the Worker ‚Äî all other requests serve static assets directly from edge cache with zero Worker invocation
- Worker uses `WorkerEntrypoint` class and `env.ASSETS.fetch(request)` to serve assets when needed
- Source: [Worker Script Routing](https://developers.cloudflare.com/workers/static-assets/routing/worker-script/)

**D1 Worker Binding API:**
- Fully typed TypeScript support via `wrangler types`
- Prepared statements with `.bind()`, `.run()`, `.first()`, `.raw()`, `.batch()`
- Returns `meta` object with `rows_read` and `rows_written` for billing tracking
- Source: [D1 Worker API](https://developers.cloudflare.com/d1/worker-api/)

### Cost Reality

At 94 page views/day, the analytics Worker would handle ~94 beacon requests/day. With 3 indexes on the table, each INSERT costs 4 rows written (1 table row + 3 index rows). That's 376 rows written/day ‚Äî **0.376% of the free tier** (100K rows written/day).

The `/api/stats` endpoint reads are similarly negligible. A dashboard query scanning 30 days of data (~2,800 rows) costs 2,800 rows read ‚Äî **0.056% of the free tier** (5M rows read/day).

Static page requests (the blog itself) are **free and unlimited** ‚Äî they never invoke the Worker.

If the blog ever exceeds free tier (100K page views/day), paid pricing is: $0.001/million rows read, $1/million rows written. At 100K views/day with 4 rows written each = 400K rows/day = 12M rows/month. Cost: $12/month. Still cheaper than Plausible ($9/mo for 10K views).

## Proposals

### Proposal A: Worker Script + D1 ‚≠ê

A client-side beacon (`<script>`) sends a `POST /api/event` on page load. The Worker script handles only `/api/*` routes ‚Äî all other requests serve static assets directly from edge cache with zero Worker invocation. The Worker writes to D1 and serves a `/api/stats` JSON endpoint for the public dashboard.

**Architecture:**
```
Static pages (free, unlimited):
  Browser ‚Üí Cloudflare Edge ‚Üí static HTML from dist/ (no Worker invoked)

Analytics beacon (Worker invoked only for /api/*):
  Browser <script> ‚Üí POST /api/event ‚Üí Worker ‚Üí D1 INSERT

Dashboard:
  /stats page ‚Üí fetch /api/stats ‚Üí Worker ‚Üí D1 SELECT ‚Üí JSON response

gkoreli.com
‚îú‚îÄ‚îÄ /*              ‚Üí static HTML (served from edge cache, free, no Worker)
‚îú‚îÄ‚îÄ /api/event      ‚Üí Worker function (receives beacon, writes to D1)
‚îú‚îÄ‚îÄ /api/stats      ‚Üí Worker function (queries D1, returns JSON)
‚îî‚îÄ‚îÄ /stats          ‚Üí static page (public dashboard, client-side fetch to /api/stats)
```

**What the Worker captures (all server-side, from `request.cf`):**
- `country`, `city`, `continent`, `region` ‚Äî geographic data, free on all plans
- `colo` ‚Äî which Cloudflare datacenter handled the request
- `timezone` ‚Äî visitor's timezone
- `asOrganization` ‚Äî ISP/cloud provider (useful for bot detection)
- URL path, referrer, user-agent ‚Äî from standard request headers
- Visitor hash: `hash(IP + UA + daily_salt)` ‚Äî unique counting without storing PII
- Owner exclusion: opt-out cookie or `?me` param skips logging

**Why this is the right architecture:**
- Static pages are never slowed down ‚Äî the Worker only runs on `/api/*` routes
- The beacon is a tiny `POST` ‚Äî doesn't block page rendering
- D1 writes are fast (indexed INSERT takes <1ms per Cloudflare docs)
- Geographic data comes free from Cloudflare's edge ‚Äî no third-party GeoIP service needed
- Full SQL means any dashboard query we can imagine

**Billing impact per page view:**
- 1 Worker invocation (the beacon POST) ‚Äî counts against 100K/day free tier
- 4 D1 rows written (1 table + 3 indexes) ‚Äî counts against 100K/day free tier
- 0 cost for the static page itself ‚Äî free and unlimited

### Proposal B: Worker Script + Analytics Engine

Same approach but write to Analytics Engine instead of D1.

**Pros:**
- Fire-and-forget writes via `writeDataPoint()` (zero latency impact)
- Purpose-built for event ingestion

**Cons:**
- **90-day data retention on free plan** ‚Äî lose historical data permanently
- Limited SQL subset ‚Äî can't do complex queries for the dashboard
- Schemaless (blobs/doubles/indexes) ‚Äî harder to reason about than explicit SQL schema
- Less community examples, newer product

### Proposal C: Cloudflare Web Analytics Only

Already enabled. Keep as supplementary performance monitor.

**Proven limitations (from actual usage):**
- Can't distinguish owner from visitors (42 of 43 "visits" were the owner)
- Counts sessions not people ‚Äî refreshing = new visit
- No public dashboard, no API
- Performance-focused (Core Web Vitals), not content-focused
- Black box ‚Äî can't filter, query, or export

### Proposal D: Plausible Cloud ($9/mo)

Evaluated and rejected.

- $9/mo for page view counts on a personal blog with ~100 views/day is overkill
- Don't own the data ‚Äî locked in their dashboard
- Self-hosted Plausible costs the same ($5-10/mo VPS) with more maintenance
- We already have free serverless infrastructure that can do this
- No way to customize the public dashboard to match the blog's design

## Decision

**Proposal A (Worker + D1)** as the primary analytics system. **Proposal C (Web Analytics)** stays enabled as a supplementary performance monitor.

### Why D1

- Full SQL ‚Äî build any query, any dashboard, any export
- Unlimited retention ‚Äî see trends over months and years (vs Analytics Engine's 90 days)
- Explicit schema ‚Äî easy to reason about and evolve
- Owner exclusion ‚Äî the #1 problem with Web Analytics, solved trivially
- Free at our scale ‚Äî 100K rows written/day vs our ~376 rows/day (0.376%)
- Time Travel ‚Äî 7-day point-in-time recovery on free plan, automatic backups

### Why Not Analytics Engine

- 90-day retention kills long-term trend analysis
- Limited SQL means we'd hit querying walls building the dashboard
- Schemaless data model is harder to reason about than explicit SQL

### Why Not Plausible

- $9/mo for something we can build in a few hours on free infrastructure
- Don't own the data
- Can't customize the public dashboard to match the blog's design
- Philosophy: build it yourself, understand how it works

### Data Schema (D1)

```sql
CREATE TABLE page_views (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  path TEXT NOT NULL,
  referrer TEXT,
  country TEXT,
  city TEXT,
  continent TEXT,
  visitor_hash TEXT,  -- hash(IP + UA + daily_salt), no PII stored
  is_bot INTEGER DEFAULT 0,
  is_owner INTEGER DEFAULT 0,
  created_at TEXT DEFAULT (datetime('now'))
);

-- Indexes add 1 extra row written each per INSERT (3 indexes = 3 extra rows)
-- At ~100 views/day: 400 rows written/day = 0.4% of free tier (100K/day)
CREATE INDEX idx_path ON page_views(path);
CREATE INDEX idx_created ON page_views(created_at);
CREATE INDEX idx_visitor ON page_views(visitor_hash);
```

Note: D1 bills per row read/written, not per query. Each index adds 1 extra row written per INSERT on the indexed column. With 3 indexes, each page view INSERT costs 4 rows written total. This is documented in [D1 Pricing ‚Äî Definitions](https://developers.cloudflare.com/d1/platform/pricing/#definitions), point 6.

### Worker Config

```jsonc
// wrangler.jsonc ‚Äî additions to existing config
{
  "name": "gkoreli-com",
  "main": "src/worker/index.ts",
  "assets": {
    "directory": "packages/blog/dist",
    "binding": "ASSETS",
    "not_found_handling": "404-page",
    "html_handling": "auto-trailing-slash",
    "run_worker_first": ["/api/*"]
  },
  "d1_databases": [
    { "binding": "DB", "database_name": "blog-analytics", "database_id": "TBD" }

```

Key config decisions:
- `run_worker_first: ["/api/*"]` ‚Äî Worker only runs for API routes. All static pages served directly from edge cache, free and unlimited.
- `binding: "ASSETS"` ‚Äî needed so the Worker can call `env.ASSETS.fetch(request)` if we ever need to serve assets through the Worker (e.g., for HTMLRewriter transforms).
- `not_found_handling: "404-page"` ‚Äî serves `404.html` for unknown routes (existing behavior).

### Worker Script Pattern (TypeScript)

```typescript
// src/worker/index.ts ‚Äî simplified pattern from Cloudflare docs
import { WorkerEntrypoint } from "cloudflare:workers";

interface Env {
  DB: D1Database;
  ASSETS: Fetcher;
}

export default class extends WorkerEntrypoint<Env> {
  async fetch(request: Request): Promise<Response> {
    const url = new URL(request.url);

    if (url.pathname === "/api/event" && request.method === "POST") {
      return this.handleEvent(request);
    }
    if (url.pathname === "/api/stats") {
      return this.handleStats();
    }

    // Fallback: serve static assets
    return this.env.ASSETS.fetch(request);
  }
}
```

### Client Beacon Pattern

```html
<!-- Lightweight beacon in page template, ~200 bytes -->
<script>
  fetch('/api/event', {
    method: 'POST',
    body: JSON.stringify({ path: location.pathname, referrer: document.referrer }),
    headers: { 'Content-Type': 'application/json' },
    keepalive: true
  });
</script>
```

`keepalive: true` ensures the beacon completes even if the user navigates away. The beacon is non-blocking ‚Äî it doesn't await the response.

### Privacy

**Architecture: Server-side daily-salted hash ‚Äî no cookies, no browser storage, no fingerprinting.**

Our unique visitor counting uses: `hash(daily_salt + IP + user_agent)`

This is the industry-standard privacy-first pattern for cookieless analytics, verified in the actual source code of four major projects:

**Plausible** (21K stars, Elixir) ‚Äî `lib/plausible/ingestion/event.ex` line 565:
```elixir
SipHash.hash!(salt, user_agent <> request.remote_ip <> domain <> root_domain)
```
Salt is a 16-byte cryptographic random value (`lib/plausible/session/salts.ex`), stored in a `salts` database table, rotated via an Oban worker (`lib/workers/rotate_salts.ex`). Old salts are cleaned up after 48 hours. They keep the previous salt to handle session continuity across rotation boundaries.

**Umami** (23K stars, TypeScript) ‚Äî `src/app/api/send/route.ts` lines 108-110:
```typescript
const sessionSalt = hash(startOfMonth(createdAt).toUTCString());
const sessionId = uuid(sourceId, ip, userAgent, sessionSalt);
```
Uses SHA-512 hash (`src/lib/crypto.ts`), salt derived from start-of-month timestamp. Session ID is a UUIDv5 of `hash(websiteId + ip + userAgent + monthlySalt)`. Note: Umami rotates monthly, not daily ‚Äî longer tracking window but same principle.

**WP Statistics** (600K+ WordPress installs, PHP) ‚Äî `includes/class-wp-statistics-ip.php`:
```php
$dailySalt = ['date' => $date, 'salt' => hash('sha256', wp_generate_password())];
// ... later:
return hash('sha256', $dailySalt['salt'] . $ip . $userAgent);
```
SHA-256 hash with daily salt stored in WordPress options table. Salt regenerated when date changes. The hash is truncated and prefixed with `#hash#` for identification.

**Wide Angle Analytics** (commercial, [documentation](https://wideangle.co/documentation/cookieless-tracking)):
Uses daily salt + IP + user-agent hash. Session ID is "additionally randomized with a daily salt value" so "a visit from the same browser, from the same website, and from the same IP address will be unique only within the duration of a single day."

**Key implementation differences across projects:**

| Project | Hash Algorithm | Salt Rotation | Inputs | Storage |
|---------|---------------|---------------|--------|---------|
| Plausible | SipHash | ~24h (Oban job) | salt + UA + IP + domain + root_domain | PostgreSQL `salts` table |
| Umami | SHA-512 ‚Üí UUIDv5 | Monthly | websiteId + IP + UA + monthlySalt | In-memory |
| WP Statistics | SHA-256 | Daily (date check) | dailySalt + IP + UA | WordPress options |
| **Ours** | **SHA-256** | **Daily (midnight UTC)** | **dailySalt + IP + UA** | **D1 or env var** |

All four use the same core pattern: `hash(rotating_salt + IP + user_agent)`. The differences are in hash algorithm, rotation frequency, and where the salt is stored. Our approach aligns most closely with WP Statistics (daily SHA-256) and Plausible (daily rotation, domain-scoped).

**Why this is more privacy-respecting than Counterscale's `If-Modified-Since` trick:**

Counterscale uses HTTP cache headers (`Last-Modified` / `If-Modified-Since`) to track sessions without cookies. While clever, this is technically a form of **supercookie** ‚Äî using browser cache as a persistent identifier. This technique has been known since 2011 ([nikcub.me: "Persistent and Unblockable Cookies Using HTTP Headers"](https://nikcub.me/posts/persistant-and-unblockable-cookies-using-http-headers/)) and is classified alongside ETag tracking as a browser fingerprinting vector by the [Chromium security team](https://www.chromium.org/Home/chromium-security/client-identification-mechanisms/).

The CNIL (French data protection authority) explicitly addresses this in their [2021 guidance on cookie alternatives](https://www.cnil.fr/en/alternatives-third-party-cookies-what-consequences-regarding-consent): techniques that access or store information on the user's terminal equipment require prior consent under the ePrivacy Directive, regardless of whether cookies are used. The `Last-Modified` header persists in the browser cache and can be read back ‚Äî this is terminal storage by another name.

Our hash approach avoids this entirely:
- **Nothing stored in the browser** ‚Äî no cookies, no localStorage, no cache manipulation
- **Nothing persisted across days** ‚Äî the daily salt rotation means the same visitor gets a different hash tomorrow
- **Nothing reversible** ‚Äî SHA-256 is one-way; you cannot derive the IP from the hash
- **Server-side only** ‚Äî the client script sends a beacon; all privacy logic runs on the Worker

**Known trade-off:** We cannot track returning visitors across days. A visitor who comes Monday and Tuesday counts as 2 unique visitors. This is the same limitation documented by [WP Statistics](https://wp-statistics.com/resources/counting-unique-visitors-without-cookies/), [Plausible](https://news.ycombinator.com/item?id=25457889), and [Matomo](https://seresa.io/blog/data-loss/the-cookieless-tracking-myth-what-you-actually-lose/). For a personal blog, this is an acceptable trade-off ‚Äî we care about "how many people visited today" not "is this the same person from last week."

**GDPR/CCPA compliance:**
- IP addresses are personal data under GDPR. We never store them ‚Äî only the irreversible hash.
- The hash with daily salt rotation is the approach Plausible uses to claim GDPR compliance without consent banners. Source: [Plausible blog](https://plausible.io/blog/google-analytics-cookies), [HN discussion](https://news.ycombinator.com/item?id=42273230).
- No consent banner needed ‚Äî we don't access or store anything on the user's terminal (no cookies, no cache, no localStorage). This satisfies the ePrivacy Directive's terminal access rules as interpreted by the CNIL.
- No third-party scripts or tracking pixels from external services
- Geographic data comes from Cloudflare's edge network (`request.cf`), not from client-side APIs
- The `/stats` page itself is the transparency mechanism ‚Äî everything we collect is visible publicly

## Prior Art (Source Code Analysis)

Two open source projects exist for exactly this use case on Cloudflare Workers. We read their actual source code to extract patterns, best practices, and anti-patterns.

### Counterscale ‚Äî [benvinegar/counterscale](https://github.com/benvinegar/counterscale) (2K stars, MIT, v3.4.1)

The most mature self-hosted analytics for Cloudflare. 464 commits, 24 releases, actively maintained.

**Architecture (from source):**
- Uses **Analytics Engine** (not D1) ‚Äî `writeDataPoint()` with blobs/doubles/indexes
- Full Remix dashboard UI for querying Analytics Engine via SQL API
- R2 long-term storage (Apache Arrow files) to work around 90-day retention
- Multi-site via `data-site-id` attribute
- Monorepo: `packages/tracker` (client), `packages/server` (Worker + dashboard)

**Tracker patterns (from `packages/tracker/src/`):**

1. **Script auto-discovery** (`tracker.ts`): The `<script>` tag finds itself via `document.getElementById("counterscale-script")`, reads `data-site-id` and derives the collect URL from its own `src` attribute (`script.src.replace("tracker.js", "collect")`). Smart ‚Äî no hardcoded URLs.

2. **GET not POST for beacons** (`lib/request.ts`): Counterscale uses `XMLHttpRequest` GET requests to `/collect?p=/path&h=host&r=referrer&sid=site`, not `fetch()` POST. The response is a **1x1 transparent GIF** (`R0lGODlhAQABAIAAAAAAAP///yH5BAEAAAAALAAAAAABAAEAAAIBRAA7`). This is the classic tracking pixel pattern ‚Äî works even when JavaScript `fetch` is blocked by ad blockers.

3. **Cookieless unique visitor counting** (`analytics/collect.ts`): Uses `If-Modified-Since` / `Last-Modified` HTTP headers as a session mechanism. The server sets `Last-Modified` to today's date with seconds encoding hit count (1=new visit, 2=anti-bounce, 3+=regular). Browser caches this and sends it back on next request. No cookies, no localStorage, pure HTTP caching semantics. Source: [notes.normally.com/cookieless-unique-visitor-counts](https://notes.normally.com/cookieless-unique-visitor-counts/).

4. **Bounce detection** (`analytics/collect.ts`): Hit count 1 = bounce (visitor left after one page). Hit count 2 = anti-bounce (cancels the bounce). Hit count 3+ = normal. This is tracked via the `Last-Modified` seconds trick above.

5. **SPA navigation tracking** (`lib/instrument.ts`): Monkey-patches `history.pushState` and listens for `popstate` events to auto-track client-side navigation. Preserves original function arity (2 params) per [Sentry's wrapping guide](https://blog.sentry.io/wrap-javascript-functions/#preserve-arity). Returns a cleanup function.

6. **Referrer extraction** (`shared/utils.ts`): Checks multiple sources in order: explicit referrer param ‚Üí `document.referrer` ‚Üí URL query params (`ref`, `referer`, `referrer`, `source`, `utm_source`). Strips query strings from referrer URLs. Filters out self-referrals (same hostname).

7. **UA parsing on server** (`analytics/collect.ts`): Uses `ua-parser-js` server-side to extract browser name, version, device type, device model. Masks browser version for privacy (`maskBrowserVersion`). Classifies `device.type === undefined` as "desktop" (UA parser quirk).

8. **Country from `request.cf`** (`analytics/collect.ts`): Passes `context.cloudflare.cf` (which contains `country`) to the collect handler. Confirms our approach ‚Äî geographic data is free from Cloudflare's edge.

9. **UTM parameter tracking** (`shared/request.ts`): Captures `utm_source`, `utm_medium`, `utm_campaign`, `utm_term`, `utm_content` from URL params. Sends as short keys (`us`, `um`, `uc`, `ut`, `uco`) to minimize beacon size.

10. **1000ms timeout** (`lib/request.ts`): XHR requests have a 1-second timeout. If the collect endpoint is slow, the tracker gives up silently. Analytics should never block the user experience.

### yestool/analytics_with_cloudflare ‚Äî [yestool/analytics_with_cloudflare](https://github.com/yestool/analytics_with_cloudflare) (140 stars, MIT)

Simpler reference implementation using **Workers + D1 + Hono**.

**Architecture (from source):**
- Hono for routing with CORS middleware
- D1 with two tables: `t_website` (multi-site) and `t_web_visitor` (page views)
- Client script sends POST to `/api/visit`, server returns PV/UV counts in response
- Client injects counts into DOM elements by ID (`page_pv`, `page_uv`)

**D1 schema (`schema.sql`):**
```sql
t_website: id, name, domain, create_at, update_at
t_web_visitor: id, website_id, url_path, referrer_domain, referrer_path, visitor_ip, create_at, update_at
```

**Patterns from source:**

1. **Hono routing** (`src/index.ts`): Clean `app.post('/api/visit', ...)` pattern with typed bindings `Hono<{ Bindings: Bindings }>`. CORS middleware via `app.use('/api/*', cors())`. Much cleaner than raw `if/else` on pathname.

2. **IP from Cloudflare header** (`src/index.ts`): Uses `c.req.header('CF-Connecting-IP')` for visitor IP. This is the raw IP ‚Äî they store it directly in the database.

3. **Inline PV/UV response** (`src/index.ts`): The `/api/visit` endpoint both records the visit AND returns current PV/UV counts in the same response. Client script then writes counts into `<span id="page_pv">` elements. Interesting for showing live counters on the page.

4. **D1 prepared statements** (`src/lib/dbutil.ts`): Uses `db.prepare(sql).bind(...args).run()` pattern. Gets last insert ID via separate `SELECT last_insert_rowid()` query.

### Anti-Patterns to Avoid

From studying both codebases, these are patterns we should NOT follow:

1. **‚ùå Storing raw IP addresses** (yestool): `visitor_ip` column stores the actual IP. This is PII, violates GDPR, and is unnecessary. Counterscale's approach (cookieless via HTTP caching) or our approach (hashed IP + daily salt) is better.

2. **‚ùå Awaiting the beacon response** (yestool): Their client script `await`s the fetch response to get PV/UV counts. This blocks page interactivity. Counterscale's fire-and-forget XHR is better. Our `fetch()` with `keepalive: true` and no `await` is the right pattern.

3. **‚ùå Injecting meta tags into `<head>`** (yestool): Their tracker appends `<meta>` tags with author attribution into `document.head.innerHTML`. This is invasive and can break other scripts. Never modify the host page's DOM beyond what's explicitly requested.

4. **‚ùå Multi-site complexity when you have one site** (both): Counterscale has `siteId` everywhere, yestool has a `t_website` table with domain lookups. We have one site. Skip the abstraction.

5. **‚ùå Full Remix/React dashboard for simple stats** (Counterscale): Their dashboard is a full React app with routing, auth, charts. For a personal blog with ~100 views/day, a static HTML page with a few `fetch()` calls is sufficient.

### Best Practices to Adopt

1. **‚úÖ GET request + tracking pixel response** (Counterscale): Return a 1x1 GIF with `Cache-Control: no-cache`. Works even when `fetch()` is blocked. Consider this as a fallback or primary pattern.

2. **‚úÖ Cookieless unique visitor counting via HTTP caching** (Counterscale): The `If-Modified-Since` / `Last-Modified` trick is elegant ‚Äî no cookies, no localStorage, browser handles it natively. We should evaluate this vs our hash approach. Trade-off: the hash approach works across devices (server-side), the caching approach is per-browser but truly zero-storage.

3. **‚úÖ Derive collect URL from script src** (Counterscale): `script.src.replace("tracker.js", "collect")` means zero configuration. The tracker script knows where to send data based on where it was loaded from.

4. **‚úÖ Short parameter keys in beacon** (Counterscale): `p` for path, `h` for host, `r` for referrer, `sid` for site ID. Minimizes beacon URL length.

5. **‚úÖ Server-side UA parsing** (Counterscale): Parse user-agent on the server, not the client. Keeps the client script tiny. `ua-parser-js` is 17KB ‚Äî that's server-side only.

6. **‚úÖ Hono for Worker routing** (yestool): Cleaner than raw `if/else`. Typed bindings, middleware support, CORS built-in. Consider for our `/api/*` routes.

7. **‚úÖ Silent failure** (both): Analytics should never throw errors that affect the user. Both projects swallow errors gracefully. Our beacon must do the same.

8. **‚úÖ `defer` attribute on script tag** (Counterscale): Ensures the tracker loads after the page content. Never block rendering for analytics.

### What Our Library Adds (That Neither Has)

- **Owner exclusion** ‚Äî opt-out cookie or `?me` param, `is_owner` flag in D1
- **Public `/stats` page** ‚Äî neither project exposes stats publicly on the blog itself
- **D1 with unlimited retention** ‚Äî Counterscale's Analytics Engine loses data after 90 days
- **`request.cf` geographic enrichment** ‚Äî country, city, continent, region, timezone, all server-side
- **Framework-agnostic** ‚Äî works with any static site on Cloudflare Workers, not just our blog
- **Zero dependencies** ‚Äî no Hono, no ua-parser-js, no Remix. Pure Cloudflare Workers API.

### Scope: A Reusable Micro-Library

This doesn't need to be blog-specific. The pattern is universal: any static site on Cloudflare Workers that wants lightweight, privacy-first, self-hosted analytics with D1. The library could be:

- **Client**: `<script>` tag (~200 bytes) or ES module import ‚Äî sends beacon to `/api/event`
- **Worker**: `/api/event` (ingestion) + `/api/stats` (query) ‚Äî ~150 lines of TypeScript
- **Schema**: One `page_views` table with indexes ‚Äî `wrangler d1 execute` to set up

Could eventually be published as a package (e.g., `@nisli/analytics` or standalone), but for now it lives in the blog repo and serves our needs.
