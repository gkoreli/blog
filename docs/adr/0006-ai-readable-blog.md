# ADR-0006: AI-Readable Blog — llms.txt, Content API, AI Traffic Intelligence

## Status

Proposed — 2026-03-07

## Context

ADR-0004 deployed cookieless analytics with AI agent classification. ADR-0005 deployed a public `/stats` dashboard. The blog already distinguishes human, bot, and AI visitors — but only stores a type flag (0/1/2), not which agent. And while the blog is transparent to humans (public stats), it's opaque to AI agents — no structured way to discover or consume content.

This ADR proposes the third phase of the transparency-first architecture: make the blog a first-class citizen for both human readers and AI agents.

### The Vision

```
Phase 1 (ADR-0004): Collect data transparently     — cookieless, open-source
Phase 2 (ADR-0005): Show data transparently         — public /stats dashboard
Phase 3 (ADR-0006): Serve content transparently      — llms.txt, content API, AI traffic intelligence
```

The blog's ethos is "Prompted by human, generated by AI." The analytics should reflect both audiences. Humans get the stats page. AI agents get `llms.txt` + structured API. Both are first-class.

### What We Have Today

**AI classification** (`packages/analytics/src/classify.ts`):
- 15 named AI crawlers in regex: GPTBot, ClaudeBot, Claude-Web, CCBot, ChatGPT, Amazonbot, Applebot-Extended, Bytespider, TikTokSpider, GoogleOther, Google-CloudVertexBot, Meta-ExternalAgent, DuckAssistBot, PetalBot, PerplexityBot, Cohere-AI
- Stored as `visitor_type = 2` (AI) — but we don't store WHICH agent
- Dashboard shows "AI Reads" as a single number

**Content metadata** (`packages/blog/src/lib/frontmatter.ts`):
- PostMeta: `{ title, date, description, tags, slug, promptCount }`
- Markdown source files in `packages/blog/posts/`
- Build pipeline processes all posts, has full metadata at build time

**What's missing**:
- No `/llms.txt` — AI agents have no structured way to discover content
- No `.md` endpoints — AI agents must parse HTML to read posts
- No content API — no programmatic access to post index
- No AI agent breakdown — "AI Reads: 5" but not "3 from GPTBot, 2 from ClaudeBot"
- No AI traffic trend — can't see if AI consumption is growing

### What llms.txt Is

A proposed standard (llmstxt.org, by Jeremy Howard, Sep 2024) — like `robots.txt` but for helping LLMs use a website at inference time. Markdown file at `/llms.txt` with:
- H1: site name
- Blockquote: short summary
- Sections with H2 headers containing file lists (links + descriptions)
- Optional section for secondary content that can be skipped in short contexts

Used by: Cloudflare docs, Anthropic docs, FastHTML, Answer.AI, many developer sites. Cloudflare's version links to per-product llms.txt files and a `llms-full.txt` for bulk indexing.

The spec also proposes serving `.md` versions of pages at the same URL with `.md` appended — clean markdown for LLM consumption instead of HTML.

### What the Open Source Projects Do

**Plausible**: No llms.txt. No AI traffic breakdown. No content API. Dashboard is React SPA — not AI-readable.

**Counterscale**: No llms.txt. No AI awareness at all. Admin-only dashboard.

**Nobody** in the analytics space tracks AI traffic by agent name as a first-class feature. This is a genuine gap.

## Proposals

### Proposal A: llms.txt + Markdown Endpoints (Content Layer)

Generate at build time:
1. `/llms.txt` — site description, post index with titles/dates/descriptions, API endpoints, source code link
2. `/llms-full.txt` — full content of all posts concatenated (for large-context models)
3. `/{slug}.md` — clean markdown version of each post (raw source, no HTML)

All generated by the existing build pipeline — we already have all the metadata and markdown source. Zero runtime cost (static files served by Cloudflare).

### Proposal B: AI Traffic Intelligence (Analytics Layer)

1. Extract AI agent name from UA at classification time (already have the regex)
2. Store in new `agent_name TEXT` column on `page_views` (NULL for humans/bots)
3. New API endpoint or extension: `by_agent` in stats response
4. New dashboard section: "AI Agents" — ranked list showing which agents read what

Schema migration: `ALTER TABLE page_views ADD COLUMN agent_name TEXT`

### Proposal C: Content API (Programmatic Access)

`GET /api/posts` — JSON index of all posts:
```json
[
  {
    "slug": "the-agentic-product-engineer",
    "title": "The Agentic Product Engineer",
    "date": "2026-02-05",
    "description": "...",
    "tags": ["ai", "engineering"],
    "url": "/the-agentic-product-engineer/",
    "markdown": "/the-agentic-product-engineer.md"
  }
]
```

Generated at build time as a static JSON file. No Worker involvement. AI agents and tools can programmatically discover and consume content.

## Decision

**All three proposals, in order: A → B → C.**

They're complementary layers of the same vision:
- A makes content discoverable by AI agents (llms.txt)
- B makes AI consumption measurable (agent tracking)
- C makes content programmatically accessible (API)

Together: AI agents find us via llms.txt, consume content via .md endpoints, and we track which agents read what. Full transparency loop.

### Why This Order

A (llms.txt) is the foundation — without discoverability, there's nothing to measure. It's also the lowest effort (build-time generation, static files). B (AI tracking) requires a schema migration and analytics code changes — medium effort but high value. C (content API) is a static JSON file — trivial once A is done.

## Anti-Patterns

1. **❌ Dynamic llms.txt generation**: llms.txt should be static, generated at build time. Don't serve it from a Worker — it's content metadata, not dynamic data. Same reason we don't generate HTML at runtime.

2. **❌ Storing full UA string for AI agents**: UA strings are long and redundant. Extract and store just the agent name ("GPTBot", "ClaudeBot"). The classification regex already identifies them — just capture the match.

3. **❌ Separate AI analytics table**: Don't create a new table for AI traffic. Add `agent_name` column to existing `page_views`. The data model is the same — a page view is a page view, whether from a human or AI agent. The `visitor_type` + `agent_name` columns together give full classification.

4. **❌ Blocking AI agents then tracking them**: Some sites block AI crawlers via robots.txt then complain about AI traffic. We do the opposite — we welcome AI agents, serve them optimized content, and transparently track their usage. Consistency.

5. **❌ llms-full.txt with HTML**: The full content file should be clean markdown, not HTML. AI agents parse markdown better than HTML. We already have the markdown source — don't convert HTML back to markdown.

6. **❌ Content API as a Worker endpoint**: `GET /api/posts` should be a static JSON file (`dist/api/posts.json`), not a Worker route. It's build-time data. Static files are cached by Cloudflare CDN, cost nothing, and never fail.

## Gotchas

1. **llms.txt must be at root path**: `/llms.txt`, not `/api/llms.txt` or `/docs/llms.txt`. The spec requires root path. Cloudflare Workers with `assets` binding serves static files from `dist/` — so `dist/llms.txt` → `gkoreli.com/llms.txt`.

2. **Markdown endpoints need trailing slash handling**: Blog uses `/{slug}/` (trailing slash, `index.html`). The `.md` endpoint should be `/{slug}.md` (no trailing slash, direct file). Both patterns coexist in Cloudflare's asset serving — `dist/the-agentic-product-engineer/index.html` and `dist/the-agentic-product-engineer.md`.

3. **Agent name extraction from regex**: The current `AI_CRAWLERS` regex uses `|` alternation but doesn't capture groups. To extract the agent name, either use named capture groups or match against individual patterns. Named groups are cleaner: `/(?<agent>GPTBot|ClaudeBot|...)/i` then `match.groups.agent`.

4. **D1 ALTER TABLE is not atomic**: D1 migrations run as SQL statements. `ALTER TABLE ADD COLUMN` is safe for adding nullable columns — existing rows get NULL. No data migration needed. But there's no transactional DDL in D1 — if the migration fails mid-way, manual cleanup is needed.

5. **llms-full.txt size**: Concatenating all posts could produce a large file. For a blog with 1-5 posts, this is fine (<50KB). At 100+ posts, consider truncating to recent posts or splitting by topic. The spec's "Optional" section is designed for this — secondary content that can be skipped.

6. **AI agent UA strings evolve**: New AI agents appear regularly. The `AI_CRAWLERS` regex needs periodic updates. Consider: log unclassified UAs that look AI-like (heuristic: no standard browser engine string) for manual review.

7. **Cache invalidation for llms.txt**: llms.txt is generated at build time and deployed as a static file. It's automatically updated on every deploy. But Cloudflare CDN caches static files — after deploy, the old version may be served for up to the TTL. For a blog that deploys infrequently, this is fine. For frequent deploys, consider cache-busting headers.

8. **Markdown source vs rendered markdown**: Post source files have frontmatter (YAML between `---` delimiters). The `.md` endpoint should serve clean markdown without frontmatter — just the content. Strip frontmatter at build time.

## Best Practices

1. **✅ Build-time generation for all content metadata**: llms.txt, llms-full.txt, posts.json, .md files — all generated by the build pipeline. Zero runtime cost. The build pipeline already has all post metadata.

2. **✅ Agent name as a derived field**: Extract agent name from UA at write time (in `handleEvent`), not at query time. Avoids regex matching on every stats query. Store once, query cheaply.

3. **✅ Nullable column for agent_name**: Only AI visitors (visitor_type=2) have an agent_name. Humans and bots get NULL. Don't waste storage on empty strings.

4. **✅ llms.txt links to .md endpoints**: The llms.txt file lists posts with links to their `.md` versions, not HTML. AI agents follow these links to get clean markdown. The HTML versions are for humans.

5. **✅ Same build pipeline, more outputs**: Don't create a separate build step for AI content. Extend the existing `buildHTML()` function to also generate llms.txt, .md files, and posts.json. One pipeline, multiple output formats.

6. **✅ Transparency footer on /stats links to llms.txt**: The stats page already has "Analytics collected without cookies..." footer. Add a link to llms.txt — "AI agents: see our llms.txt". Meta-transparency.

7. **✅ Version the content API**: `GET /api/posts` returns a JSON array. If the schema changes, add a `version` field or use content negotiation. For now, the schema is simple enough that versioning is premature.

## Implementation Sketch

### Phase A: llms.txt + Markdown Endpoints

**Files to create/modify:**
- `packages/blog/src/templates/llms.ts` — generate llms.txt and llms-full.txt content
- `packages/blog/src/pipeline/build.ts` — add llms.txt, llms-full.txt, .md files, posts.json to build output
- No new dependencies. No Worker changes. No client JS.

**Build output additions:**
```
dist/llms.txt              — site index for AI agents
dist/llms-full.txt         — full content concatenation
dist/api/posts.json        — JSON post index
dist/{slug}.md             — clean markdown per post (no frontmatter)
```

### Phase B: AI Traffic Intelligence

**Files to modify:**
- `packages/analytics/src/classify.ts` — return agent name alongside type
- `packages/analytics/src/db.ts` — add agent_name to PageView interface and INSERT
- `packages/analytics/src/stats.ts` — add `by_agent` to StatsResponse and query
- `packages/analytics/schema.sql` — add agent_name column
- `packages/blog/src/client/stats.ts` — add "AI Agents" section to dashboard
- `packages/blog/src/templates/stats.ts` — add skeleton for AI Agents section
- D1 migration: `ALTER TABLE page_views ADD COLUMN agent_name TEXT`

### Phase C: Content API

Already covered by Phase A — `dist/api/posts.json` is generated at build time.

## Adjacent Ideas (Future)

Ideas that emerged during exploration but are out of scope for this ADR:

- **Reading engagement**: Scroll depth + time on page via beacon events. Would answer "do people actually read the posts or just skim?" Requires new event types in the beacon, new D1 columns, new dashboard section.
- **Data lifecycle**: Retention policy (archive rows older than N days to R2), data export (CSV download from /stats). Operational necessity as D1 grows.
- **Period comparison**: "This week vs last week" on the dashboard. Requires a second API call and delta calculation in the client.
- **RSS analytics**: Track RSS feed consumption. The feed is already generated — add a tracking pixel or use Cloudflare analytics on the feed URL.
- **Structured data (JSON-LD)**: Add schema.org Article markup to blog posts. Helps search engines and AI agents understand content structure. Build-time generation.
